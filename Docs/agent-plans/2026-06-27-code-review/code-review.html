Overall this is a small, well-organized AppKit application with a clear separation between the simulation core (Grid), the rendering/interaction layer (GridView), the document/format readers and writers, and the preferences/library UI. The simulation engine itself is sound: it double-buffers each generation, recomputes population correctly, and supports configurable born/survive rule sets and cell aging for the color display.

The most serious problems are concentrated in two areas: concurrency in the library loading path (an unbalanced dispatch group that can hang library loading, and unsynchronized array mutation from multiple threads), and untrusted input handling when opening .gol and .rle files (overflow/huge-allocation crashes, a logically-inverted magic-number check, a divide-by-zero from a zero cell size, and a range crash in the RLE decoder). There is also a hot-loop busy-wait on the main thread during pattern drag-and-drop, and two genuine latent bugs (a 16-vs-32 bit shift in readUInt64, and a missing S prefix that silently drops the "Day & Night" rule). None of these block normal use of the core simulation, but several are reachable from ordinary user actions (opening a file, dragging a pattern, editing the "draw every" field).

H1 — Library load can hang permanently on a failed entry

High
concurrencystabilityhang
GameOfLife/Classes/LibraryReader.swift:56–74

Each library entry calls dispatch.enter() and dispatches work asynchronously, but the worker returns early on failure without calling dispatch.leave(). dispatch.wait() then blocks forever for that group.

for i in p.value {
    dispatch.enter()
    DispatchQueue.global( qos: .userInitiated ).async {
        guard let items = self.load( object: i ) else {
            return                       // <-- leaves the group unbalanced
        }
        group.allChildren.append( contentsOf: items )
        group.children.append( contentsOf: items )
        dispatch.leave()
    }
}
dispatch.wait()                          // <-- never returns if any entry failed

load(object:) returns nil for any malformed entry, or any include:/patterns: reference that fails to resolve. Because reload() in LibraryViewController runs this on a background queue and only sets loading = false after read() returns, a single bad entry leaves the spinner running indefinitely and the library never appears. A user editing Library.json via the built-in editor can trigger this directly.

Recommendation: Ensure leave() is always balanced with enter() — e.g. defer { dispatch.leave() } as the first statement inside the closure, or restructure with dispatch.notify instead of a blocking wait().

H2 — Data race: concurrent mutation of the same arrays during library load

High
concurrencyundefined behaviorcrash
GameOfLife/Classes/LibraryReader.swift:60–72

Multiple closures are dispatched concurrently to a global (concurrent) queue, and each one appends to the same group.allChildren and group.children Swift arrays:

DispatchQueue.global( qos: .userInitiated ).async {
    ...
    group.allChildren.append( contentsOf: items )   // unsynchronized
    group.children.append( contentsOf: items )      // unsynchronized
    dispatch.leave()
}

Array is not thread-safe. Concurrent append from several threads is undefined behavior — it can corrupt the array's backing storage and crash anywhere in the process. This path runs at startup (library view is shown by default), so it is exercised on essentially every launch; it likely "works" most of the time only because contention is low.

Recommendation: Collect each worker's result into a thread-local value and merge on a serial queue, or guard the appends with a lock / serial queue, or use concurrentPerform writing into pre-sized index slots. Combine the fix with H1.

H3 — Main-thread busy-wait and data race when dropping a pattern

High
concurrencyperformancehang
GameOfLife/Classes/GridView.swift:632–637 · LibraryItem.swift:173–199

performDragOperation spins in an empty while loop on the main thread, waiting for rotations that are computed asynchronously on a background queue:

while item.rotations.count < self.draggedRotation
{}                                  // busy-wait on the main thread

cells = item.rotations[ self.draggedRotation - 1 ]

Two problems: (1) item.rotations is written on a background queue in prepareRotations with no synchronization, so this is a data race; without a memory barrier the loop is not even guaranteed to observe the updated count, and it pins a CPU core at 100% while it waits. (2) prepareRotations finishes by calling DispatchQueue.main.sync; if rotation computation is still in flight while the main thread is spinning here, the design relies on the unsynchronized write becoming visible to break the loop — fragile and, in the worst case, a hang.

Recommendation: Make rotation preparation synchronous at drop time, or have prepareRotations hand the finished rotations back through a completion handler / thread-safe accessor and perform the drop from that callback. Never busy-wait on the main thread.

M1 — A crafted .gol file can crash the app (integer overflow / huge allocation)

Medium
securitystabilitycrash
GameOfLife/Classes/Grid.swift:283–353

load(data:) reads width and height as 64-bit values straight from the file and uses them to size allocations without bounds checking:

cells.reserveCapacity( Int( width * height ) )
...
data.advanced( by: 44 ).decompress( with: COMPRESSION_LZFSE,
                                     bufferSize: Int( width * height ) )

width * height is an unchecked UInt64 multiplication that traps on overflow, and Int(width * height) traps if the product exceeds Int.max. Even short of overflow, a moderately large product drives UnsafeMutablePointer.allocate / reserveCapacity to allocate gigabytes, causing an OOM crash. Since .gol files can be opened directly from Finder, a malformed or hostile file crashes the app on open.

Recommendation: Validate width and height against sane maxima (and against the actual payload size) before multiplying or allocating; use overflow-checked arithmetic (multipliedReportingOverflow(by:)) and bail out with a clean failure.

M2 — Inverted boolean logic in the .gol magic-number check

Medium
correctnesssecurity
GameOfLife/Classes/Grid.swift:290

The header signature check uses && where it must use ||:

if( data[ 0 ] != 71 && data[ 1 ] != 79 && data[ 2 ] != 76 && data[ 3 ] != 49 )
{
    return false
}

The intent is to reject the file unless the first four bytes are exactly "GOL1" (71, 79, 76, 49). As written, the file is rejected only if all four bytes are wrong; any file that matches even one byte (e.g. starts with G) passes validation and proceeds into parsing. This weakens the guard and lets clearly-invalid files reach the allocation logic in M1.

Recommendation: Use || so the check rejects the file if any signature byte differs from the expected value.

M3 — A .gol file with cellSize 0 causes a divide-by-zero crash

Medium
stabilitycrash
GameOfLife/Classes/Grid.swift:301,345 · GridView.swift:60,123–124,339–341

load(data:) only rejects a stored cell size greater than 10 (if size > 10 { return false }) and then writes it straight into preferences: Preferences.shared.cellSize = UInt( size ). A size of 0 is accepted. Every place that derives grid dimensions then divides by cellSize:

var width  = size_t( self.frame.size.width  / self.cellSize )   // cellSize == 0
var height = size_t( self.frame.size.height / self.cellSize )

With cellSize == 0 this yields infinity, and converting a non-finite CGFloat to size_t traps. The same division happens in GridView.init and the drawing loop, so the app crashes shortly after opening such a file.

Recommendation: Reject (or clamp) a stored cell size of 0 in load(data:), and defensively guard the divisions in GridView against a zero cell size.

M4 — RLE decoder crashes on a zero or over-large run count

Medium
stabilitycrash
GameOfLife/Classes/RLEReader.swift:216–231

When handling a row terminator ($), the decoder builds a half-open range from the run count minus one:

let i = ( n == "" ) ? 1 : Int( n ) ?? 0
for _ in 0 ..< i - 1 { lines.append( "" ) }

If the run count is 0 (an explicit "0$") or fails to parse — for example an extremely long digit string where Int(n) overflows to nil and falls back to 0 — then i == 0 and the expression becomes 0 ..< -1. Forming a Range with upperBound < lowerBound traps at runtime. As .rle files are an opened document type, a malformed pattern crashes the import.

Recommendation: Validate the parsed count (treat 0/overflow as 1 or skip), and guard the loop with if i > 1 before constructing the range.

M5 — "Draw every N generations" field accepts 0 → modulo-by-zero crash

Medium
stabilitycrash
GameOfLife/Classes/GridView.swift:254–256 · GOL.PreferencesWindowController.xib:111–122

The animation timer computes whether to redraw with a modulo on the draw interval:

let skip = Preferences.shared.drawInterval
if( self.paused || self.grid.turns % UInt64( skip ) == 0 ) { self.display() }

If drawInterval is 0, turns % 0 traps. While the slider in the preferences XIB enforces a minimum of 1, the bound text field ("Draw every: [ ] Generation") uses a number formatter with no minimum (minimumIntegerDigits="0", no minimum), so a user can type 0 directly into the field. Once the simulation is running, this crashes the app.

Recommendation: Clamp drawInterval to a minimum of 1 in the setter / binding (or set a formatter minimum), and guard the modulo with max(1, skip).

M6 — "Day & Night" rule is silently dropped (missing "S" prefix)

Medium
correctness
GameOfLife/Classes/Rule.swift:51

The rule string is missing the S before the survive digits:

Rule( name: "B3678/34678",  title: "Day & Night", predefined: true ),  // should be B3678/S34678

The failable initializer requires the second component to begin with "S" (Rule.swift:92); since "34678" begins with 3, the initializer returns nil and availableRules() filters it out. The "Day & Night" rule therefore never appears in the rule list, contrary to intent.

Recommendation: Fix the literal to "B3678/S34678".

L1 — readUInt64 shifts the high word by 16 instead of 32

Low
correctnesslatent bug
GameOfLife/Classes/Data.swift:58–61

The 64-bit reader combines its two 32-bit halves with the wrong shift:

public func readUInt64( at: Int ) -> UInt64 {
    return UInt64( self.readUInt32( at: at ) ) << 16     // should be << 32
         | UInt64( self.readUInt32( at: at + 4 ) )
}

The high 32 bits must be shifted left by 32, not 16, to sit above the low word. This is currently masked because every value written to a .gol header (version, width, height, cell size, compression flag) fits in 32 bits, so the high word is always 0 and the result happens to be correct. It is nonetheless a real bug that would corrupt any value ≥ 2³². Note the sibling readUInt32 is correct (it shifts its high half by 16).

Recommendation: Change the shift to << 32.

L2 — RulerView can spin in an infinite loop

Low
stabilityhang
GameOfLife/Classes/RulerView.swift:127–154

Inside the line-numbering loop, two continue branches skip the statements that advance range.location:

guard let rectArray = layoutManager.rectArray( ... ) else { continue }  // range not advanced
if( rectCount == 0 ) { continue }                                      // range not advanced

If either condition holds, the loop condition is unchanged and it never terminates, hanging the UI while editing the library JSON. These cases are uncommon (they require the layout manager to return no rects for a glyph range) but are reachable for certain text states.

Recommendation: Advance range.location (e.g. to end) before continue, or restructure so the loop always makes progress.

L3 — RLEWriter force-unwrap crashes on a zero-width grid

Low
stabilitycrash
GameOfLife/Classes/RLEWriter.swift:111

After the inner column loop, the writer force-unwraps old:

rle += ( ( n > 1 ) ? String( describing: n ) : "" ) + ( ( old! & 1 == 1 ) ? "o" : "b" )

If grid.width == 0 while grid.height > 0, the inner loop never executes, old stays nil, and this force-unwrap traps. A zero-width grid is unusual but can arise from the default Grid(width: 0, height: 0) path or odd resize states, and "Save As… (RLE)" would then crash.

Recommendation: Guard against an empty grid up front, or handle the old == nil case instead of force-unwrapping.

L4 — Library editor assigns the vertical ruler view twice

Low
correctnessdead code
GameOfLife/Classes/LibraryEditorWindowController.swift:40–44

A RulerView is created and assigned, then immediately overwritten by a second, different instance:

let ruler = RulerView()
self.textView?.enclosingScrollView?.verticalRulerView = ruler
self.textView?.enclosingScrollView?.verticalRulerView = RulerView()   // discards `ruler`
self.textView?.enclosingScrollView?.verticalRulerView?.clientView = self.textView

The local ruler is wasted, and the intent (which instance should be installed) is unclear. Harmless at runtime but confusing.

Recommendation: Keep a single instance — assign ruler once and set its clientView.

L5 — Grid.next allocates a buffer it may immediately discard

Low
performance
GameOfLife/Classes/Grid.swift:108–115

The new generation buffer is grown before the turns == UInt64.max guard, so on the (admittedly unreachable in practice) saturation case the work is wasted:

cells.grow( self.cells.count ) { Cell() }
if( self.turns == UInt64.max ) { return }

Minor; reorder so the early-return precedes the allocation. Also note that on saturation the simulation silently freezes with no user feedback.

Recommendation: Move the turns == UInt64.max check above the allocation.

L6 — add() can under-count population for aged-but-dead cells

Low
correctness
GameOfLife/Classes/Grid.swift:430–438

When stamping a pattern, population is incremented only when the target cell value is exactly 0:

if( self.cells[ i ] == 0 ) { self.population += 1 }
self.cells[ i ] = 1

A cell that is dead but still carries age bits in the high bits (value even and non-zero) is not equal to 0, so the increment is skipped even though the cell transitions from dead to alive — under-counting population. After next() dead cells are always exactly 0, so this is normally fine, but cells loaded from a raw .gol payload may carry such values.

Recommendation: Test the alive bit (self.cells[ i ] & 1 == 0) rather than the whole byte when deciding to increment.

I1 — Duplicate rule definition (Replicator and Fredkin)

Info
correctness
GameOfLife/Classes/Rule.swift:41–42

"Replicator" and "Fredkin" are both defined as B1357/S02468. They are distinct named rules in the literature (Fredkin is typically B1357/S02468 and Replicator B1357/S1357), so at least one name maps to the wrong rule set. Both will appear in the list with identical behavior.

Recommendation: Confirm the intended rule strings and correct one of the two entries.

I2 — Significant duplicated code

Info
maintainability
GridView.swift:56–90 · RLEReader.swift:29–58 / CellReader.swift:29–58 · Array.swift / ContiguousArray.swift

Several blocks are near-verbatim copies: the three GridView initializers repeat identical grid-creation/observer/registration setup; RLEReader.read(directory:) and CellReader.read(directory:) are identical except for the file-reader call; and Array.grow / ContiguousArray.grow are byte-identical. The drag-handling teardown in draggingExited/draggingEnded is also duplicated. This duplication invites divergent bug fixes over time.

Recommendation: Extract shared helpers (a common GridView setup method, a generic directory-scanning helper parameterized by the per-file reader, and a single generic grow on RangeReplaceableCollection).

I3 — Fragile reflection-driven Preferences and forced casts

Info
stabilitymaintainability
GameOfLife/Classes/Preferences.swift:58–67,103–108

Preferences.init mirrors over its own stored properties and calls setValue(_:forKey:) with whatever UserDefaults returns. If a future non-optional property is added without a matching entry in Defaults.plist, object(forKey:) returns nil and KVC will crash trying to set a non-optional. Separately, color(index:) uses a forced cast self.value(forKey:) as! UInt? that would trap if the stored type ever differs. These work today (the plist covers all current keys) but are brittle.

Recommendation: Skip nil values when registering, or assert that each reflected key has a registered default; replace the forced cast with a safe as? and a fallback.

Strengths

  • Clean simulation core. Grid.next() correctly double-buffers (reads the old grid, writes a fresh one), recomputes population in the same pass, and cleanly factors the rule into bornSet/surviveSet so arbitrary B/S rules and the cell-aging color scheme drop out naturally.
  • Sensible persistence. The .gol format carries a signature and an optional LZFSE-compressed payload with a graceful fallback to uncompressed when compression doesn't help (Grid.data()).
  • Good model hygiene. LibraryItem implements NSSecureCoding with typed, guarded decoding and a proper pasteboard read/write path.
  • Consistent, readable style. The whole codebase follows one formatting convention, with small focused types (mostly one per file) and clear separation of concerns between model, view, and controllers.
  • Defensive option handling at the UI edges. Most controller code uses guard let with early returns rather than force-unwrapping outlets.

Appendix — Severity guide

  • Critical Exploitable or guaranteed data loss / crash on common paths.
  • High Serious correctness/concurrency/stability defect reachable from ordinary use.
  • Medium Real bug with a meaningful but narrower trigger (often malformed input or a specific field).
  • Low Minor or latent issue; limited impact or hard to trigger today.
  • Info Maintainability / clarity observation, not a defect.

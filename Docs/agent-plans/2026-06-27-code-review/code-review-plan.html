This plan turns the 18 code-review findings into small, reviewable milestones grouped by theme rather than by severity, so each milestone touches a coherent set of files and can be committed on its own. The five phases move from the highest-impact, most-reachable defects (concurrency hangs and data races) through untrusted-file parsing hardening and UI input edge cases, then the remaining correctness/latent bugs, and finally the maintainability refactors.

Per the project working agreement and the user's choices for this plan: no unit tests are added as part of these milestones (manual verification only), and nothing is committed or pushed automatically — each milestone is left for manual review and commit. Every fix should be verified by building through Xcode (xcodebuild) since an .xcodeproj is present, and by exercising the affected code path manually where practical.

Phase 1 — Concurrency & hangs (High severity)

Milestone 1.1 — Fix library load hang and data race

Completed
H1H2concurrencyhangdata race
GameOfLife/Classes/LibraryReader.swift:56–74 · LibraryViewController.swift (reload/loading flag)
Goal

Stop the library loader from hanging forever when an entry fails, and eliminate the undefined-behavior data race from concurrent unsynchronized appends. The library view is shown by default, so this path runs on essentially every launch.

Tasks
  • Guarantee dispatch.leave() is always balanced with dispatch.enter() — e.g. defer { dispatch.leave() } as the first statement inside the async closure, or restructure with dispatch.notify instead of a blocking wait() (H1).
  • Remove the data race on group.allChildren/group.children: collect each worker's result into thread-local storage and merge on a serial queue, guard the appends with a lock/serial queue, or use concurrentPerform writing into pre-sized index slots (H2).
  • Confirm LibraryViewController.reload() still clears loading = false on all paths once loading completes (including the all-entries-failed case).
  • Build via Xcode and verify: a malformed Library.json entry no longer leaves the spinner running; a valid library loads completely and in stable order.
Status
Completed
Completed

2026-06-27

Implementation notes
  • H1 (hang). In LibraryReader.read(url:), the per-entry worker closure now calls dispatch.leave() from a defer block placed as the first statement, so the guard let items = self.load(object: i) else { return } early-return can no longer skip it. The dispatch.enter()/leave() pair is now always balanced and dispatch.wait() can no longer block forever when an entry fails to load.
  • H2 (data race). Removed the concurrent unsynchronized appends to group.allChildren/group.children. Each worker now writes its result into a pre-sized, index-keyed slot (results[index]), keyed by the entry's position via p.value.enumerated(). The slot write is serialized through a dedicated serial DispatchQueue, defined once as a private static let (LibraryReader.syncQueue) and reused across all groups, so concurrent writes can never race. After dispatch.wait(), the slots are merged into the group on the calling thread in order. This eliminates the race and, as a bonus, makes the load order deterministic/stable (previously order depended on thread completion timing).
  • Loading flag (task 3). Confirmed LibraryViewController.reload() already sets self.loading = false unconditionally on the main thread after reader.read(...) returns (including the empty/all-failed case, where read returns groups with empty children). No change needed there once the hang is fixed.
  • Decision. Synchronize the pre-sized index-slot writes with a dedicated serial DispatchQueue rather than an NSLock (per reviewer preference, see Review comments). The queue is defined once as a private static let on LibraryReader instead of being instantiated per group iteration. This fixes the race and yields a stable ordering with minimal structural change, matching the existing code style (no new lock types, no new dependencies).
  • Verification. Built Game Of Life - Debug via xcodebuild → BUILD SUCCEEDED. (SourceKit flagged isolated-file resolution warnings for project types such as CellReader/RLEReader; these are false positives — the full build compiles cleanly.)
Review comments
  • 2026-06-27. Reviewer dislikes the use of NSLock to synchronize the result writes — prefers a dedicated (serial) dispatch queue instead.
  • 2026-06-27. Reviewer prefers the synchronization queue to be defined once as a private static let rather than instantiated per group iteration.

Milestone 1.2 — Fix main-thread busy-wait and data race on pattern drop

Completed
H3concurrencyperformancehang
GameOfLife/Classes/GridView.swift:632–637 · LibraryItem.swift:173–199
Goal

Remove the empty while busy-wait that pins a CPU core on the main thread waiting for asynchronously-computed rotations, and remove the unsynchronized read/write of item.rotations.

Tasks
  • Replace the busy-wait in performDragOperation with a correct hand-off: either make rotation preparation synchronous at drop time, or have prepareRotations deliver finished rotations via a completion handler / thread-safe accessor and perform the drop from that callback.
  • Synchronize access to item.rotations so the writer in prepareRotations and the reader at drop time cannot race (serial queue, lock, or completion-based hand-off).
  • Ensure the drop applies the correct rotation index (draggedRotation) without relying on a write becoming visible by chance.
  • Build via Xcode and verify: dragging and dropping a rotated pattern works with no UI freeze and no CPU spin.
Status
Completed
Completed

2026-06-27

Implementation notes
  • Root cause. LibraryItem.prepareRotations computed the three rotations on a background global queue and wrote self.rotations there, while performDragOperation (drop) and drawItem (live preview) read item.rotations from the main thread. The drop spun an empty while item.rotations.count < self.draggedRotation {} busy-wait pinning a CPU core on the main thread (H3 hang), and the cross-thread unsynchronized read/write of the rotations array was a data race (potential crash via buffer refcount race), not merely a stale read.
  • Fix — make rotation preparation synchronous. Every caller (draggingEntered), every reader (drawItem, performDragOperation), and the draw pass all run on the main thread, and rotating a small library pattern three times is cheap. Removed the DispatchQueue.global(...).async / DispatchQueue.main.sync wrapper from prepareRotations so it computes inline and calls done() directly. rotations is now only ever touched on the main thread, which serializes all access inherently — no lock or queue needed, and the race is removed at its source. Dropped @escaping from done and gave it a default {} so it can be called as a no-op to force preparation.
  • Fix — remove the busy-wait (GridView.performDragOperation). Replaced the empty while spin with item.prepareRotations() (a no-op if already prepared, otherwise computes synchronously) followed by a bounds guard self.draggedRotation <= item.rotations.count else { return false } and a direct read of item.rotations[ self.draggedRotation - 1 ]. The correct rotation index is now applied deterministically on the same thread that prepared it, never relying on a write becoming visible by chance.
  • Scope note. Considered computing the rotation synchronously at drop only (leaving prepareRotations async), but that would leave the drawItem preview read still racing the background writer. Making prepareRotations synchronous fixes the drop, the preview, and the busy-wait together with the smallest change, and matches the review's first suggested option. The rotations property, its pasteboard encode/decode, and copy(with:) are otherwise unchanged.
  • Verification. Built Game Of Life - Debug via xcodebuild → BUILD SUCCEEDED. Runtime drag-drop behavior (no freeze / no CPU spin when dropping a rotated pattern) is reasoned from the code; not exercised in a launched app here.

Phase 2 — Untrusted-file parsing hardening

Milestone 2.1 — Harden .gol file loading

Completed
M1M2M3securitystabilitycrash
GameOfLife/Classes/Grid.swift:283–353 (290, 301, 345) · GridView.swift:60,123–124,339–341
Goal

Make Grid.load(data:) robust against malformed or hostile .gol files (openable directly from Finder): no overflow/huge-allocation crash, a correct signature guard, and no divide-by-zero from a zero cell size.

Tasks
  • M1: Validate width and height against sane maxima and against the actual payload size before multiplying or allocating; use overflow-checked arithmetic (multipliedReportingOverflow(by:)) and bail out with a clean failure instead of trapping or allocating gigabytes.
  • M2: Fix the inverted magic-number check at Grid.swift:290 — use || so the file is rejected if any of the four signature bytes differs from "GOL1" (71, 79, 76, 49).
  • M3: Reject (or clamp) a stored cellSize of 0 in load(data:), and defensively guard the frame / cellSize divisions in GridView (init, drawing loop) against a zero cell size.
  • Build via Xcode and verify: a crafted file with huge dimensions, a wrong signature, or cellSize == 0 fails to open gracefully instead of crashing.
Status
Completed
Completed

2026-06-27

Implementation notes
  • M2 — inverted signature check (Grid.swift:290). Changed the four byte comparisons from && to ||, so the file is now rejected if any of the "GOL1" signature bytes (71, 79, 76, 49) differs. Previously the file was only rejected when all four bytes were wrong, letting almost any file through the guard.
  • M1 — overflow / huge allocation (Grid.load(data:)). Before any allocation, width and height (both UInt64) are multiplied with width.multipliedReportingOverflow( by: height ); the load bails out cleanly on overflow or when the product exceeds a new cap, Grid.maxCellCount = 1 << 28 (268,435,456 cells = 256 MiB, since cells are one byte). The validated product is converted to Int count once and reused for reserveCapacity, the LZFSE decompress(bufferSize:) allocation, and the payload-size equality checks — so a hostile file can no longer trap on overflow nor drive a gigabyte-scale UnsafeMutablePointer.allocate in decompress. Switching the equality checks to the Int count also removes the prior mixed Int/UInt64 comparisons.
  • M3 — zero cell size. (1) In load(data:) the cell-size guard is now if( size < 1 || size > 10 ), rejecting a stored cellSize of 0 (it previously only rejected > 10) before it could reach Preferences.shared.cellSize. (2) Defensively, every GridView divisor that derives from the preference now reads CGFloat( max( 1, Preferences.shared.cellSize ) ) — this covers the three initializers and the two self.cellSize assignments (resizeGrid/updateDimensions), which in turn feed the drawing-loop and layout divisions. Clamping at the source guarantees no frame / cellSize division can be by zero (which would produce a non-finite value and trap when converted to size_t).
  • Decision — the 256 MiB cap. The review asked for "sane maxima" without a number. 1 << 28 cells is well above any realistic grid (screen size ÷ minimum cell size) yet bounds a single load allocation to 256 MiB. Flagging it here so you can adjust the constant if you'd prefer a tighter or looser bound.
  • Scope note. Left the readUInt64 shift bug (L1) alone — it is Milestone 4.1. It does not affect this path because the written width/height/cellSize all fit in 32 bits (their high word is 0), so they read back correctly today.
  • Verification. Built Game Of Life - Debug via xcodebuild → BUILD SUCCEEDED. Behavior against crafted files (huge dimensions / wrong signature / cellSize == 0 failing gracefully) is reasoned from the code; not exercised against real crafted files here.

Improvements — Implementation Plan

Generated: 2026-06-27

Source: Docs/agent-plans/2026-06-27-improvements/improvements.html

Overview

This plan turns the five-item Improvements roadmap into small, individually reviewable and committable milestones for the Game Of Life macOS app. The work spans the whole stack: it introduces an automated test safety net where there is none today, fixes a real rendering regression caused by the modern NSView clipping default, re-architects the simulation core from a fixed-size dense array into an unbounded chunked/tiled grid, modernizes the codebase into idiomatic Swift, and finally documents the resulting API with DocC comments.

The milestones are deliberately ordered to minimize wasted effort. Tests come first so they can guard every later change. The isolated drawing bug is fixed early. The largest and most architectural change — the infinite grid — is performed before the broad modernization and documentation passes, so that those polishing passes apply to the final architecture rather than to code that is about to be rewritten. Each milestone carries its own status, completion date, implementation notes, and review comments, and is updated in place as work proceeds (the /xs:implement command edits this file milestone by milestone).

Questions & answers

Which testing framework should the new unit-test target use?
Swift Testing (Apple's modern @Test/#expect framework), reaching internal types via @testable import.
How should the plan handle the Infinite Grid item, given it is the largest, most architectural change?
Full implementation, broken into several phased milestones within this plan.
Which unbounded-grid representation should the infinite-grid work target?
A chunked/tiled grid (fixed-size live tiles allocated on demand) — a good balance of speed, memory, and migration cost from the current dense array.
How should the performance-critical hot loop in Grid.next() be treated during modernization?
Modernize everywhere, including next(), accepting a possible performance cost in exchange for consistency. The test target includes a next() benchmark so any regression is at least measured and visible.
Should the Documentation Comments item also set up generated DocC reference output?
Comments only. Add /// doc comments across public and private API; treat a generated DocC reference as a possible future follow-up, not part of this plan.

Notes

  • Build & test tooling. The project is an Xcode project (GameOfLife.xcodeproj, single app target Game Of Life, schemes Debug/Release/AppStore) with no Swift package. Per the working agreement, build and run tests through xcodebuild, and keep the Xcode project the source of truth for target/membership changes.
  • Coding conventions. Follow the repository's working agreement: prefer let over var; never force-unwrap; prefer functional constructs over raw loops; early returns; one type per file, organized into subfolders; explicit public/open for API; document public and private members; align consecutive =; separate calls from assignments with a blank line.
  • Key simulation files. Grid.swift (dense ContiguousArray<UInt8> core, next() evolution, .gol data()/load() persistence, library/pattern add()), Rule.swift (B/S rule parsing and the predefined rule list), RLEReader.swift/RLEWriter.swift/CellReader.swift (pattern I/O), Data.swift (binary read helpers).
  • Key rendering files. GridView.swift (overrides draw(_:) and isFlipped), plus other custom-drawing views: ColorView, BorderView, ColorWell, ProgressIndicator, RulerView, LibraryItemView, LibraryItemDetailView. The clipping regression centers on these.
  • Cell encoding. A cell is one UInt8: bit 0 is alive/dead, the remaining high bits store the cell's age (used for color/aging). Any new grid representation must preserve this so colors and the .gol format keep working.
  • Milestone ordering deviates from the source order (which listed Infinite Grid second and Drawing last). The rationale above (don't polish or document code that is about to be rewritten; ship the isolated bug fix early; tests first) drives the reordering. If you prefer the original order, say so and the plan can be resequenced.
  • Relationship to the existing code review. A separate code-review-plan.html exists alongside the source under Docs/agent-plans/2026-06-27-code-review/. Recent commits already address several review findings (grid next() guard order, readUInt64 high-word shift, rule de-duplication, KVC hardening). This plan focuses on the roadmap improvements; where a milestone touches the same code, avoid re-litigating already-fixed review items and check the latest state of the file first.
  • Risk — infinite grid is invasive. It changes the in-memory model, the on-disk .gol format, pattern import, and rendering/scrolling simultaneously. The design milestone (M5) must settle the representation, the persistence-format strategy (versioning vs. new magic), and the rendering/viewport model before the migration milestones begin. Tests written in M2/M3 against the dense grid will partly need revision afterward; the rule-application and RLE-format tests remain valid because they are independent of the internal representation.

Review comments
  • Q: The other two schemes (Release, AppStore) don't run tests — do they need to? Investigated empirically by running xcodebuild test … -configuration Release. Tests cannot run under the Release/AppStore configs for two independent reasons: (1) SWIFT_ENABLE_TESTABILITY defaults to NO there (it is YES only in Debug), so @testable import GOL fails to build — "Unable to find module dependency: 'GOL' … swiftmodule built without '-enable-testing'"; and (2) ENABLE_HARDENED_RUNTIME = YES in those configs would block injecting the test bundle into the host via library validation. M2.Core simulation tests — grid evolution & rules

Completed
Goal

Lock down the correctness of the simulation core — the part most at risk during modernization and the infinite-grid rewrite — with focused, deterministic tests.

Tasks
  • Test Rule parsing: valid B/S strings produce the right born/survive sets; malformed strings fail the failable initializer; name is uppercased and displayTitle is formed correctly.
  • Cover the predefined rule list from availableRules() (count, uniqueness, a few spot-checked born/survive sets).
  • Test Grid.next() against known Conway results on a blank grid: still lifes (block), oscillators (blinker period 2), and a glider's displacement after four generations.
  • Verify population and turns are accurate after evolution, and that next() is a no-op once turns reaches UInt64.max.
  • Test alternate rules (e.g. Seeds, HighLife) on small hand-computed boards.
  • Test cell aging: a long-lived cell's age bits increment and saturate at the cap without affecting the alive bit.
  • Test cellAt/isAliveAt/setAliveAt bounds behavior, and resize() preserving in-bounds cells and recomputing population.
Status

Completed   Completed: 2026-06-27

Implementation notes
  • Layout. Added tests under GameOfLifeTests/Classes/ (mirroring GameOfLife/Classes/) plus a GameOfLifeTests/Support/ group: RuleTests.swift, GridEvolutionTests.swift, GridAccessorTests.swift, and GridTestSupport.swift. Files were added to the target with the xcodeproj gem. Total suite is now 22 tests in 4 suites, all green via xcodebuild test … -scheme "Game Of Life - Debug".
  • Test support. GridTestSupport provides: makeGrid([String]) building a blank grid from ASCII art ("O" = live), render(Grid) back to ASCII for readable diffs, liveCoordinates(Grid), and withActiveRule(_:_:) which sets Preferences.shared.rule for the duration of a closure and restores it after.
  • Rule parsing (RuleTests). Valid B/S parsing (single- and multi-digit sets), displayTitle/name formation, the predefined flag, and a parameterized test that malformed strings ("", "B3", "B3/S2/3", "X3/S23", "B3/X23") fail the failable initializer. Predefined list checked for count (15), all-predefined, unique names, and spot-checked born/survive sets (Conway, Seeds, HighLife).
  • Evolution (GridEvolutionTests, .serialized). Because Grid reads its rule from Preferences.shared.activeRule() (there is no per-call rule parameter), every rule-dependent test drives the global rule via withActiveRule; the suite is .serialized so concurrent tests can't leak rules into each other. Covers Conway canonical results against independent ground truth — block (still life), blinker (period-2 oscillator), and a glider verified to translate by exactly (+1, +1) after four generations — plus turns/population bookkeeping, Seeds (B2/S) and HighLife (B36/S23, isolating the B6 birth that Conway lacks), and cell aging.
  • Accessors (GridAccessorTests). Bounds behavior of cellAt/isAliveAt (including negative coordinates), setAliveAt set/clear and out-of-bounds no-op, and resize() growing (cells + population preserved) and shrinking (out-of-bounds cells dropped, population recomputed). These run unserialized as they never call next().
  • next() overflow guard. turns has a private setter, so the "no-op once turns == UInt64.max" test forces it via KVC (setValue(NSNumber(value: .max), forKey: "turns") — it is an @objc dynamic property); confirmed the poke takes effect and a further next() neither advances turns nor changes the cells.
  • Aging semantics confirmed. A cell set via setAliveAt starts at age 0 (raw byte 1); a surviving cell's raw byte is 1 | (age << 1), incrementing each generation and saturating at age 127 (byte 255) with the alive bit always set. The aging assertions were written to unwrap cellAt explicitly via #require and compare plain UInt8 values: the optional-vs-literal form (cellAt == 1 | (5 << 1)) produced a spurious failure under the full concurrent run; the unwrapped form is stable across the test run alone and 13 consecutive full-suite runs.
  • Findings (no production changes made — out of scope for a tests milestone). (1) Rule uppercases self.name but then parses the original-case input and checks the B/S prefixes case-sensitively, so a lowercase rule like "b3/s23" fails to parse and the uppercasing is effectively unobservable for valid rules. It is not a live bug because callers (activeRule()) uppercase before constructing, but it is an inconsistency worth a later look. (2) setAliveAt does not maintain population (only next()/load/add/resize do), so tests assert population only after evolution/resize. No bugs were found in next(), the rules, or aging — all canonical results matched.

(A) The #O creator line was built from a localized, style-based DateFormatter and the whole file then encoded as ASCII; on recent macOS the time contains U+202F (narrow no-break space), so data(using: .ascii) returned nil and RLE export silently failed. Fixed by formatting the date with a fixed en_US_POSIX locale and an explicit ASCII dateFormat. (B) The writer carried its run-length state (old/n) across row boundaries, inflating the next row's run count and corrupting multi-row exports. Fixed by resetting the run state at the start of each row. With both fixed, the multi-row round-trip is now a true passing test (no longer a known issue).
  • Reader quirk (documented, not fixed — out of scope). RLEReader.decode seeds its line list with an empty string, so every decoded pattern gains a leading empty row. A dedicated test documents this; round-trip tests assert live-cell content (ignoring the leading empty row).
  • Test-isolation fix (reviewer approved): run grid/Preferences tests on the main thread. The tests are hosted in the fully-launched app, whose main-window GridView animates on the main thread and observes Preferences. Mutating Preferences.shared (rule/cellSize) from background test threads fired GridView.resize() off the main thread, racing the main-thread animation over the same grid and crashing intermittently with Index out of range (this also explains the rare M2 flakiness). Per the chosen approach, the suites that create Grids or touch Preferences (WiringTests, GridEvolutionTests, GridAccessorTests, GridPersistenceTests, LibraryPlacementTests, PatternReaderTests, NextBenchmarkTests) are now @MainActor, so their bodies and the KVO reactions they trigger run on the main thread, serialized with the app UI. RuleTests and DataTests touch neither grids nor preferences and stay parallel. No production change was needed for this; the only production change in M3 is the RLEWriter fix above.

Status

Completed   Completed: 2026-06-28

Implementation notes
  • Corrected root cause. The regression is not about clipsToBounds. On recent macOS the rect passed to a view's drawing method is only the dirty region and is no longer guaranteed to equal self.bounds (previously it effectively did). Views that drew using that passed rect for their geometry now paint only part of themselves. The fix is to draw against self.bounds and ignore the passed rect — no clipsToBounds changes.
  • Audit result. Of the eight custom-drawing views, only two used the passed rect: ColorWell.draw(_:) (NSBezierPath(roundedRect: rect, …)) and RulerView.drawHashMarksAndLabels(in:) (the background rect.fill() and the label width/height from rect.size). Both now use self.bounds. The others already drew against self.bounds (ColorView, BorderView, LibraryItemView, LibraryItemDetailView), a local copy of bounds (ProgressIndicator), or self.frame and never the passed rect (GridView) — so they were left unchanged.
  • Tests (DrawingBoundsTests, @MainActor). Each view is rendered into an offscreen bitmap with a tiny corner dirty rect, then a far corner pixel is checked to confirm it was still painted. These failed before the fix (far corner alpha 0.0 — only the dirty rect drawn) and pass after (whole bounds drawn). 54 tests in 10 suites all green.
  • Note on the milestone's original premise. The milestone was written around the macOS 14 clipsToBounds default change; the actual on-device symptom here was the dirty-rect/bounds mismatch instead, so the fix and its code comments reflect that rather than setting clipsToBounds.
  • Preferences colors table margins. A separate new-macOS rendering regression: the colors NSTableView in GOL.PreferencesWindowController.xib had no explicit style, so it defaulted to .automatic, which now resolves to the inset style and indented the rows from the table's left/right edges. Fixed in the xib by setting tableStyle="fullWidth" (with selectionHighlightStyle="none") to restore edge-to-edge rows.
Review comments
  • Reviewer correction. An initial attempt set clipsToBounds = true on several views; the reviewer clarified that the real issue is the passed rect no longer equalling self.bounds, and that the fix is simply to draw using self.bounds and ignore the passed rect (no clipsToBounds). That attempt was fully reverted and the milestone redone accordingly.

Phase 3 — Infinite grid (chunked/tiled)

M5.Design the chunked/tiled grid & abstraction seam

In progress
Goal

Settle the architecture before touching the migration: how live tiles are stored and allocated, how neighbor counting works across tile boundaries, and how persistence and rendering adapt — so the implementation milestones are mechanical.

Tasks
  • Specify the tile model: fixed tile dimensions, the UInt8 cell layout (alive bit + age) preserved per cell, on-demand allocation, and reclamation of empty tiles.
  • Define the coordinate system for an unbounded plane (signed coordinates, tile keying) and how the visible viewport maps onto it.
  • Design neighbor counting across tile edges/corners (halo/ghost cells or neighbor-tile lookups) and the generation step over only populated tiles.
  • Define an abstraction seam (protocol or clear API boundary) so call sites — GridView, controllers, persistence — depend on grid behavior, not on the dense array.
  • Decide the .gol persistence strategy for unbounded grids (bump the format version / new magic vs. extend the header) and how RLE/library import maps into the new model.
  • Capture the chosen design (data structures, edge handling, format change, viewport model) in this milestone's implementation notes and get it reviewed before M6.
Status

Completed   Completed: 2026-06-28

Implementation notes
  • Decisions taken with the reviewer. (1) Navigation: pan/scroll only — the window is a movable viewport into the infinite plane at a fixed cell size; zoom is deferred to a possible later follow-up. (2) Save compatibility: read old, write new — keep reading existing fixed-size .gol files (importing their live cells), but always save in the new tiled format.
  • Tile model. Fixed 32×32 tiles, each a ContiguousArray<UInt8> of 1024 bytes; the cell encoding is unchanged (bit 0 alive, bits 1–7 age, cap 127). Tiles are allocated on demand (a cell set or born in a missing tile creates it) and reclaimed when they become all-dead, so memory tracks the live population rather than the explored area. Tile size is a single tunable constant to revisit against the M3 benchmark.
  • Coordinate system. World coordinates are signed Int (x, y), unbounded. A cell maps to tile (x.floorDiv(32), y.floorDiv(32)) at local (x.mod(32), y.mod(32)) using floored div/mod so negative coordinates work. Populated tiles live in a [TileKey: Tile] dictionary keyed by the tile's (col, row).
  • Neighbour counting & generation. Each generation processes the active set = populated tiles ∪ their 8 neighbours (births can appear in an empty tile adjacent to live cells). Each active tile's next state is computed from its own 1024 cells plus a 1-cell halo copied from the 8 neighbouring tiles (zeros where absent), keeping the inner loop local (no per-cell dictionary lookups). Considered and rejected: per-cell cross-tile lookups (simpler but slower at boundaries). Aging, population, turns, and the turns == UInt64.max no-op are preserved.
  • Abstraction seam. Grid stays the boundary type (still NSObject with KVO turns/population); only its internals change from ContiguousArray<Cell> to a tile store — a protocol would be overkill for one implementation. The leak to remove is GridView reading grid.cells[...] directly: replace with bounded query API — cellAt/isAliveAt/setAliveAt over signed coordinates plus an iteration method (e.g. forEachLiveCell(in: worldRect)) for rendering. width/height stop meaning "the playfield"; a live bounding box (computed from populated tiles) replaces them where UI needs extents, and the fixed-size resize()-to-screen logic is retired in M8.
  • Persistence (.gol v2). The current header already carries an unused UInt64 version field at offset 4 (always 0 today); bump it to 1 for the tiled format — no new magic. Read: version 0 → parse the old fixed-size grid and import its live cells at world origin; version 1 → read sparse tiles. Write: always version 1 — cellSize, tile count, then each populated tile's (col, row) + bytes, LZFSE-compressed as today, with the existing malformed/oversized guards carried over. add(cells:left:top:)/load(item:)/RLE import place into the unbounded plane with no width/height clamping.
  • Viewport / rendering (pan only). GridView holds a viewport origin (world coords of the top-left visible cell) and cellSize; draw() iterates only the visible world rect through the grid API. Pan (drag/scroll) moves the origin; editing, library insertion and the ruler convert screen↔world through that origin. No zoom.
  • Downstream split. M6 = tile store + unbounded next() + accessors behind the seam (dense-grid tests updated, glider-travels-forever test added, benchmark re-run). M7 = .gol v2 read/write + old-file import + RLE/library placement. M8 = viewport rendering, pan navigation, retire resize()-to-screen, screen↔world mapping for editing/ruler.
Review comments
  • Design approved by the reviewer (pan-only navigation; read-old/write-new saves; 32×32 tiles with halo-based neighbour counting; Grid kept as the API seam; .gol version bump from 0 to 1).

M6.Migrate the simulation core to the chunked grid

Completed
Goal

Replace the fixed-size dense array with the chunked/tiled representation so the simulation grows without edges, while keeping behavior identical for patterns that fit the old bounds.

Tasks
  • Implement the tile store and the unbounded next() generation step, expanding the active region as patterns grow.
  • Re-implement cellAt/isAliveAt/setAliveAt, population, and turns against the new model.
  • Preserve cell aging semantics (age bits increment and saturate) in the new step.
  • Keep or adapt the existing public API behind the M5 seam so non-rendering call sites compile with minimal change.
  • Update the M2 evolution tests to the unbounded model and add tests proving a glider travels indefinitely without hitting a boundary.
  • Re-run the M3 next() benchmark and record the chunked-grid figure against the dense baseline.
Status

Completed   Completed: 2026-06-28

Implementation notes
  • Approach (approved before coding). Implemented via TDD with a minimal-ripple facade: the tile store becomes the source of truth and next() goes unbounded, but the legacy dense surface (width/height/cells/resize() and the bounds-checked size_t accessors, plus the v0 .gol round-trip) is kept as a windowed facade so persistence (M7) and rendering (M8) keep working against the old fixed playfield until their own milestones retire it. A new signed-Int seam was added alongside for genuinely-unbounded access. This keeps the M6 diff small and let every pre-existing test stay green as a regression net.
  • Tile store. Grid internals changed from ContiguousArray<Cell> to [TileKey: Tile] where Tile = ContiguousArray<UInt8> of 32×32 = 1024 bytes and TileKey is a private (col, row) struct. Tiles are allocated on demand (setCellValue creates the owning tile) and reclaimed when they become all-dead. World↔tile mapping uses floored div/mod helpers (tileIndex/localIndex) so negative coordinates work; the cell encoding (bit 0 alive, bits 1–7 age, cap 127) is unchanged. Tile size is a single private constant (Grid.tileSize) to revisit against the benchmark.
  • Unbounded next(). Each generation builds the active set = populated tiles ∪ their 8 neighbours (so births can appear in an empty adjacent tile). For each active tile it fetches the 3×3 block of neighbour tiles once, builds a (32+2)² halo of alive bits (centre tile + a 1-cell border from the neighbours, zero where absent), then computes the next state from contiguous halo reads — no per-cell dictionary lookups. The rule application and aging block are copied verbatim from the dense version, so behavior is identical for in-window patterns. Output tiles with no live cell are dropped (reclamation). turns, the turns == UInt64.max no-op, and aging are preserved.
  • Accessors & the new seam. The size_t accessors (cellAt/isAliveAt/setAliveAt) keep their exact windowed, bounds-checked semantics (size_t is Swift's signed Int, so the existing x >= 0 guards and cellAt(x:-1) == nil behavior are retained) but delegate to signed tile lookups. The new unbounded seam is forEachLiveCell { x, y, cell in … }, which iterates the whole plane (including negative/out-of-window coordinates); next() and the new test use it. No accessor overloading (which would have made integer-literal call sites ambiguous).
  • Semantics worth noting. (1) population is now the total live count across the whole plane, not just the window — required so the travelling-glider test reads 5 after the glider leaves the window; for in-window patterns total == window count, so no existing test changed. (2) resize() keeps the dense model's destructive reframe (it prunes live cells outside the new window and recomputes the population), preserving the current app behavior on live-resize; it is retired in M8. (3) load(item:) deliberately keeps the original's quirk of not resetting population before placement (no behavior change vs. dense).
  • Persistence kept on v0 (M7 will change it). data()/load(data:) still read and write the existing v0 .gol format, now materialized through the windowed cells facade (write) and populated into the tile store at world origin (read), with all the malformed/oversized guards intact. The v2 tiled format, old-file migration and unbounded RLE/library placement are M7.
  • Two call sites touched (deviation from "untouched"). The plan's facade option assumed GridView/RLEWriter would be untouched by reading grid.cells, but cells is now a computed window snapshot, so indexing it inside their per-cell draw/export loops would be O(n²) (a real freeze, not just slower). Both single read lines were changed to the O(1) accessor (grid.cellAt(x:y:) ?? 0) instead — a one-line edit each, no logic change. Flagged here for review; the alternative was an unusably slow app across the M6/M7 commits. The cells facade is still used by data().
  • Raw loops retained (deferred to M9). The new next()/tile code uses raw for loops, matching the existing simulation core and the perf-sensitive inner loop. M9 is the dedicated milestone for converting loops (explicitly including the next() hot loop) to functional style and re-measuring, so loop-modernization was intentionally not pulled into M6.
  • Tests. RED→GREEN: added gliderTravelsUnbounded ("a glider travels far beyond the old fixed bounds") to GridEvolutionTests — it seeds a glider on a 10×10 window, runs 40 generations (a +10,+10 shift well past where the old dense 10×10 grid would have destroyed it at the edge), and asserts via forEachLiveCell that it landed at the translated position with population 5. The existing M2 evolution, accessor, resize, persistence, library-placement and RLE tests all remained valid unchanged (they exercise patterns within the window) and stay green, confirming behavior is preserved for patterns that fit the old bounds. Full suite: 55 tests in 10 suites, all green.
  • Benchmark (re-run, Debug/-Onone). Chunked grid on a fully-random 256×256: ~45 ms/gen (44.8–46.1 across three runs) vs. the M3 dense baseline of ~18 ms/gen (itself noisy, 18–54 ms/gen). A fully-populated square is the worst case for the tile model — every tile and its neighbours are active and the halo build/dictionary overhead is paid on every cell, none of which the flat dense array paid. Sparse patterns (the actual use case for an infinite grid) touch far fewer tiles and cost much less; M9 will modernize the loops and re-measure.

Status

Completed   Completed: 2026-06-28

Implementation notes
  • Format design (decided with the reviewer, extends the M5 spec). The reviewer asked to also persist the scene/viewport, so v1 carries a scene block beyond the M5 "cellSize + tiles" sketch. We store the logical viewport (origin + extent in world cells), not the macOS window geometry, which is screen-dependent and belongs to AppKit state restoration. The fields are reserved now (filled with sensible defaults) and become fully live in M8 — so M8 needs no further format bump.
  • .gol v1 layout (big-endian, 76-byte header). magic "GOL1" (unchanged, no new magic); UInt64 version = 1; UInt64 cellSize; UInt64 tileSize; scene block = Int64 originX, Int64 originY, UInt64 viewWidth, UInt64 viewHeight; UInt64 tileCount; UInt64 LZFSE flag. Payload (LZFSE-compressed as before): one record per populated tile = Int64 col + Int64 row + tileSize² cell bytes. Signed coordinates are written via UInt64(bitPattern:) (no new Data helpers needed). For M7 the scene origin is written as (0, 0) and viewWidth/Height as the current width/height window; M8 will write the real viewport.
  • Read dispatch & migration. load(data:) validates magic, reads the version, and dispatches: version 0 → the legacy fixed-size reader (loadVersion0), importing its live cells into the plane at the world origin and setting the window to the file's width/height — so existing .gol files still open; version 1 → loadVersion1; any other version → reject. Writing is always v1 (read-old / write-new, per M5).
  • Tile-size independence. v1 read places cells by world coordinate using the file's tileSize (world = col*tileSize + lx), then re-tiles through setCellValue into the build's current tiles. So a file written with one tile size still loads correctly if the build's tile size ever changes. Verified by a hand-built test using tileSize 2.
  • Guards (carried over and extended). cellSize ∈ 1…10; tileSize ∈ 1…2¹⁶; tileCount · tileSize² ≤ 2²⁸ (the existing max-cell cap) checked with overflow-reporting multiplies; the payload length must equal tileCount · (tileSize² + 16) (post-decompress for LZFSE, exact for raw). Truncated/oversized/malformed streams are rejected before allocation, mirroring the v0 guards.
  • Unbounded placement. add(cells:left:top:) no longer clamps to width/height nor skips negative coordinates — patterns place at their world coordinate anywhere on the plane (drag-drop near an edge keeps the overflow instead of destroying it; visible after M8 pan). load(item:) is unchanged: it still centers/grows within the current view, which is the right "place it where the scene is" behavior and generalizes to M8's viewport.
  • RLE export over the live region. RLEWriter now walks the live bounding box (via a new Grid.liveBounds()) instead of the legacy window, so a pattern entirely outside the window is still exported; the run-length logic is otherwise unchanged. RLEReader needed no change (it builds a LibraryItem placed through the now-unbounded add/load(item:)).
  • Tests (TDD, RED→GREEN). Added: unbounded add placement outside the window and at negative coordinates (LibraryPlacementTests, replacing the old "clamps cells outside the grid" test); v1 round-trip preserving cells outside the window and at negative coords, a hand-built uncompressed v1 read (tileSize 2), and a truncated-payload reject (GridPersistenceTests); RLE export of cells outside the window (PatternReaderTests). Updated compressedRoundTrip to read the LZFSE flag at its new v1 offset (68). A test-support helper liveWorldCoordinates reads the whole plane via forEachLiveCell. The pre-existing v0 hand-built tests now double as old-file migration coverage. Full suite: 60 tests in 10 suites, all green.
